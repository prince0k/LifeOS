## 4. Feature Modules Specifications

---

### 4.1 Dashboard Module (MOD-Dashboard)
- **Purpose:** Provide a centralized dashboard displaying the active shift, current timetable task, recovery state, quick-log buttons, and recommendation cards.
- **Inputs:** Active shift template data, current active task, Recovery State, Recovery Score, current habit logs for the day.
- **Outputs:** Visual rendering of gauges, timers, recommendation text, and quick-log event triggers.
- **Dependencies:** MOD-Planner, MOD-Recovery, MOD-Habits.
- **Rules:**
  - `RULE-DASHBOARD-001`: If the Daily Recovery Check-in has not been completed, display a prominent check-in alert card at the top.
  - `RULE-DASHBOARD-002`: The "Quick Log (+1 Cigarette)" button must execute in one tap without opening any confirmation screens.
- **Acceptance Criteria:**
  - Tap on the "+1 Cigarette" button increments the count on the dashboard immediately.
  - Tapping the Recovery card redirects the user to the Recovery Check-in screen.

---

### 4.2 Planner Module (MOD-Planner)
- **Purpose:** Manage, load, and adapt the daily timetable based on shifts, recovery, and manual modifications.
- **Inputs:** Shift template selection, Daily Recovery State, manual task schedule shifts.
- **Outputs:** Adaptive Daily Timetable (start/end times, durations, task assignments).
- **Dependencies:** MOD-Recovery, MOD-Tasks, MOD-Settings.
- **Rules:**
  - `RULE-PLANNER-001`: Loading a shift template pre-populates the schedule:
    - **Morning Shift (10:30 AM – 6:30 PM):** Primary focus on shift duties + Mailing; Deep Work scheduled in the evening; Sleep target: 11:00 PM.
    - **Night Shift (7:30 PM – 3:30 AM):** Primary focus on Mailing before shift; Deep Work scheduled in the afternoon; Sleep target: immediately after shift.
    - **12-Hour Shift (12:00 PM – 12:00 AM):** Recovery focus; no mandatory project work; prioritize health and sleep.
    - **Off Day (No work shift):** Highest productivity; long Deep Work blocks; weekly review and planning.
  - `RULE-PLANNER-002`: Manual override must allow the user to drag, drop, resize, or delete any timetable block.
- **Acceptance Criteria:**
  - Changing the active shift template automatically rebuilds the daily schedule layout.
  - Incomplete tasks automatically rollover to the next day's backlog.

---

### 4.3 Task Module (MOD-Tasks)
- **Purpose:** Manage personal, admin, and project-related tasks.
- **Inputs:** Task title, category, priority (High, Medium, Low), associated project ID.
- **Outputs:** Categorized lists of active and completed tasks.
- **Dependencies:** None.
- **Rules:**
  - `RULE-TASK-001`: Tasks must be groupable by project (Mailing, CityHost) or standard tags (Personal, Admin).
  - `RULE-TASK-002`: High-priority tasks must be highlighted in the planner view.
- **Acceptance Criteria:**
  - Tapping a task checkbox marks it completed, moves it to the completed section, and updates the Daily Consistency Score.

---

### 4.4 Habit Module (MOD-Habits)
- **Purpose:** Track smoking increments, screen usage statistics, and custom positive habits.
- **Inputs:** Tap increments, Detailed log details (time, trigger, mood, notes), Android Usage Stats API telemetry.
- **Outputs:** Daily habit completion logs, total screen time counts, app-specific usage metrics.
- **Dependencies:** Android OS permission (Usage Stats).
- **Rules:**
  - `RULE-HABIT-001`: Smoking tracking supports two entry paths: **Quick Log** (one tap, registers $+1$ cigarette with current timestamp) and **Detailed Log** (registers count along with trigger [Stress, Boredom, Friends, Work, Craving, Habit, Other], mood, notes, and optional location).
  - `RULE-HABIT-002`: Screen time tracking automatically imports daily durations for Instagram, YouTube, Chrome, and WhatsApp via the Android Usage Stats API. The user can manually edit or input these values.
- **Acceptance Criteria:**
  - Tapping "+1 Cigarette" increments count in under 50ms locally.
  - Re-running screen time sync pulls active application usage metrics without duplicate entry creation.

---

### 4.5 Mailing Project Module (MOD-Mailing)
- **Purpose:** Track tasks, hours worked, and targets for the primary professional project.
- **Inputs:** Task updates, log hours duration, weekly targets.
- **Outputs:** Mailing task lists, progress charts, total weekly hours compiled.
- **Dependencies:** MOD-Tasks, MOD-Analytics.
- **Rules:**
  - `RULE-PROJECT-001`: Tasks must categorise into: SEO, Web Development, Website Improvements, Technical Research, Social Media Content, Documentation, Daily Planning, or Weekly Goals.
  - `RULE-PROJECT-002`: Mailing receives absolute prioritization when loading project tasks on Morning and Night shifts.
- **Acceptance Criteria:**
  - Time tracking records add to the weekly sum immediately.
  - Weekly targets update when a task status shifts to completed.

---

### 4.6 CityHost Project Module (MOD-CityHost)
- **Purpose:** Track tasks, events, hours, and creative design assets for the secondary community project.
- **Inputs:** Task updates, event dates, logged campaign hours.
- **Outputs:** Event logs, campaign tracking lists, progress metrics.
- **Dependencies:** MOD-Tasks, MOD-Analytics.
- **Rules:**
  - `RULE-PROJECT-003`: Tasks must categorise into: Video Editing, Social Media, Event Planning, Website Development, Marketing, Community Growth, Creative Design, or Content Creation.
  - `RULE-PROJECT-004`: CityHost is set as secondary priority and should only suggest tasks on Off Days or after primary Mailing goals are met.
- **Acceptance Criteria:**
  - Up-coming campaigns list in chronological order of their event date.

---

### 4.7 Sleep & Recovery Modules (MOD-Sleep, MOD-Recovery)
- **Purpose:** Conduct adaptive recovery check-ins, compute the daily Recovery Score, and determine the Recovery State.
- **Inputs:** Manual Sleep Check-in (Start/Wake times, quality, wake-ups), energy (1-10), stress (1-10), mood, activity checks, Health Connect integrations (optional).
- **Outputs:** Recovery Score (0-100), Recovery State string, recommended schedule capacity limit.
- **Dependencies:** MOD-Planner, MOD-Settings.
- **Rules:**
  - `RULE-RECOVERY-001` (Adaptive Check-In): The check-in form only prompts for data not already known (e.g., if sleep was imported via Health Connect, do not ask for sleep times; if a walk was completed and logged, do not ask about the walk).
  - `RULE-RECOVERY-002` (State Triggers): Scores trigger states: Fully Recovered ($\ge 80$), Good Recovery ($70$-$79$), Moderate Recovery ($60$-$69$), Low Recovery ($50$-$59$), Recovery Needed ($40$-$49$), Burnout Risk ($< 40$).
- **Acceptance Criteria:**
  - Check-in takes under 30 seconds to complete.
  - Sleep duration calculates correctly accounting for night wake-ups (duration minus wake-ups * 15m).

---

### 4.8 Analytics Module (MOD-Analytics)
- **Purpose:** Process logged items and display correlation graphs and trends.
- **Inputs:** Historical habit logs, consistency scores, recovery history.
- **Outputs:** Correlation metrics, chart representations.
- **Dependencies:** Hive DB.
- **Rules:**
  - `RULE-ANALYTICS-001`: Aggregations must run locally. External API calls for data analysis are strictly prohibited.
- **Acceptance Criteria:**
  - Displays correlations between stress and smoking count without lag.

---

### 4.9 Journal & Brain Dump Modules (MOD-Journal, MOD-BrainDump)
- **Purpose:** Allow user to record raw text dumps and process them into actionable items.
- **Inputs:** Raw multi-line text input, processing targets.
- **Outputs:** New Tasks, Daily Note logs, or Project tasks.
- **Dependencies:** MOD-Tasks, MOD-Mailing, MOD-CityHost.
- **Rules:**
  - `RULE-BRAINDUMP-001`: One-tap processing allows converting text lines directly into tasks, moving them to projects, appending to journals, or discarding them.
- **Acceptance Criteria:**
  - Processing a line deletes it from the Brain Dump box and creates the chosen target item successfully.

---

### 4.10 Settings Module (MOD-Settings)
- **Purpose:** Manage global preferences, shift templates, and data backups.
- **Inputs:** Shift template parameters, backup trigger, imported JSON backup file.
- **Outputs:** Encrypted JSON data export file, configuration file updates.
- **Dependencies:** Hive DB.
- **Rules:**
  - `RULE-SETTINGS-001`: Data backups must execute locally. The app must prompt native Android file share dialog to allow user to save or share the backup file.
- **Acceptance Criteria:**
  - Backup files restore the application state perfectly to the identical state prior to backup.

---

## 5. Edge Cases
- **Database Lock on Import:** If the database is locked during a restore operation, the import must close all open Hive boxes, overwrite the storage folder, and restart the Flutter runner process safely.
- **Conflicting Shift Assignments:** If the user logs both "Night Shift" and "Morning Shift" on the same calendar day, the system defaults to the template that has the earliest start time.
